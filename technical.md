# Technical Architecture

How this project is actually built: the two-repo split, the build pipeline, what every script does, and where a CSS/JS/markup change belongs. This is architecture reference, not process ‚Äî for how we (human + AI) actually work together (the requirements-doc loop, git rhythm, milestone tracking), see `way-of-working.md`. For account/domain/hosting facts, see `operations.md`.

## Shape of the system

- Two sites share one codebase: the public site (`site/`) and the admin site (`admin/`) ‚Äî see `requirements/public.md` and `requirements/admin.md` for what each does, and *Two-repo structure* below for where each lives.
- No runtime backend. Every page ‚Äî including the homepage ‚Äî is a **build artifact**: a Python script embeds that page's data as JS constants directly into the HTML at build time, so pages open via `file://` or a static host with nothing to serve and nothing to keep alive.
- No npm, no bundler, no framework. `worldcup/scripts/build.py` (and the build scripts it calls) is the one build step, and is the intentional exception to that rule ‚Äî it's plain Python 3 with no dependencies.
- `.claude/launch.json` (committed, not gitignored) defines two local static-server configs ‚Äî `2026` (serves `site/` on an auto-assigned port) and `admin` (serves the repo root on port 8766) ‚Äî for AI-tool browser-preview integrations to serve pages over HTTP during a session. This doesn't change anything above: pages still open directly via `file://` with nothing to serve, this is just a convenience for previewing in a browser sandbox that can't load `file://` URLs.

## Two-repo structure

This project uses two GitHub repos under the `sportsontheinternetbydavid` account:

| Repo | Visibility | Purpose |
|------|------------|---------|
| `sports-on-the-internet-by-david-admin` | Private | Everything ‚Äî source data, scripts, docs, worldcup/shared.js/css, build tooling. This is where all work happens. |
| `sports-on-the-internet-by-david` | Public | Only the built `site/` output. What the world sees. No source files, no scripts, no docs. |

**What goes in the public repo:** the contents of `site/` only ‚Äî HTML pages, flags, nothing else. Think of it as the website, not the project.

**What stays private:** `worldcup/data/`, `worldcup/scripts/`, `worldcup/shared.js`, `worldcup/shared.css`, `worldcup/nav.css`, `requirements/public.md`, `requirements/admin.md`, `way-of-working.md`, `admin/`, all docs. Anything that isn't a finished published page.

**Deploying:** run `worldcup/scripts/deploy.py` to push the current `site/` contents to the public repo. Do this after a build when you're happy with the result. `set_result.py`, `update_day.py`, `set_knockout_size.py`, and `set_bracket_game.py` are the automatic triggers ‚Äî each commits/pushes to the admin repo and deploys to the public repo by default after updating data, via the shared `worldcup/scripts/gitops.py` helper (pass `--no-push` to opt out and update local files only). `set_team_elo.py` and `fetch_results.py` do not push ‚Äî they only update local files and rebuild.

**Remotes in the private repo:**
- `origin` ‚Üí `sports-on-the-internet-by-david-admin` (private, push here normally)
- `public` ‚Üí `sports-on-the-internet-by-david` (public, push via deploy script only)

**Admin site (future):** per `requirements/admin.md`, the admin site is local-only for now and has no deployment target. When it gets one, it should follow the same pattern as above ‚Äî its own repo, pushed via its own deploy step, never mixed into either repo above. It must never end up in the public repo.

## Scripts

Every script lives in `worldcup/scripts/`, is plain Python 3 with no dependencies, and supports `-h`/`--help`. Grouped by role:

**Data entry** ‚Äî the normal path for changing tournament data. Each writes to a `data/YEAR.json` file, then calls `build.py`. Most also then commit/push to the private repo and deploy `site/` to the public repo by default (pass `--no-push` to opt out) ‚Äî the exceptions are `set_team_elo.py` and `fetch_results.py`, which only update local files and have no push step at all.

| Script | Purpose |
|--------|---------|
| `set_result.py` | Enter a game's score (and ELO change). The everyday command ‚Äî normally generated for you by the admin site's row-click panel. |
| `set_team_elo.py` | Set a team's starting ELO for a tournament, before any results are entered. |
| `set_knockout_size.py` | One-time per-tournament setup: pick the knockout bracket size (32/16/8/4); tags whichever rounds already exist in the data and scaffolds the rest. |
| `set_bracket_game.py` | Set a knockout game's date and/or participants (a concrete team, or a "winner/loser of an earlier game" reference) ‚Äî normally generated by the admin site's Knockout Bracket section. |
| `update_day.py` | Manual fallback: bulk-enter a whole day's scores when the eloratings.net TSV hasn't caught up yet. |
| `fetch_results.py` | Daily automated fetch for the live 2026 tournament ‚Äî pulls eloratings.net's TSV and fills in any newly-available scores. |

**Build** ‚Äî regenerate HTML from data. Called automatically by every script above; run directly only when you've hand-edited a data file or changed a page template.

| Script | Purpose |
|--------|---------|
| `build.py` | Regenerates every public World Cup page + `history.html` from `data/*.json`, embedding data as JS constants. Also calls `build_home.py` and `build_admin.py` at the end, so one run keeps all three in sync. |
| `build_home.py` | Regenerates the homepage (`../site/index.html`) from a small nav config ‚Äî a build artifact like every other public page, not hand-edited, so its shared nav component (`nav.py`/`../nav.css`) has one implementation instead of a second hand-maintained copy. |
| `build_admin.py` | Regenerates the admin site's per-year pages ‚Äî same embed-at-build-time pattern as `build.py` (no server, no runtime fetch). |
| `knockout.py` | Not a script ‚Äî shared knockout-bracket round-shape logic (bracket size ‚Üí round labels/game counts), imported by `build.py`, `set_knockout_size.py`, and `set_bracket_game.py`. |
| `nav.py` | Not a script ‚Äî shared fused-nav-bar rendering (Level 1 on every World Cup page, and the homepage's whole nav), imported by `build.py` and `build_home.py`. |

**Infrastructure** ‚Äî used internally by the scripts above; you shouldn't need to run these directly.

| Script | Purpose |
|--------|---------|
| `gitops.py` | Shared commit/push/deploy helper: commits `data/`, `site/`, and `admin/` to the private repo, then calls `deploy.py`. |
| `deploy.py` | Pushes the current `site/` contents to the public GitHub Pages repo. |
| `teams.py` | Not a script ‚Äî shared team-lookup helpers (`load_teams`/`list_teams`/`resolve_team` against `data/teams.json`), imported by `set_result.py`, `set_team_elo.py`, and `set_bracket_game.py`. |

## Where CSS and JS changes go

The rendering logic is split across five layers:

- **`worldcup/nav.css`** ‚Äî the fused Level 1-3 nav's visual rules (the `.page-nav`/`.utility-bar`/`.view-toggle`/`.page-toggle`/`.primary-tabs`/`.detail-bar` system). Shared by *every* public page, including the homepage ‚Äî changes here affect the homepage and all World Cup pages simultaneously. Page-specific Level 3 variants (Rankings, Knockout, etc.) stay in `shared.css` instead.
- **`worldcup/shared.css`** ‚Äî CSS shared across every World Cup page only (base styles, table layout, page-specific Level 3 variants, Rankings view styles). Changes here affect all years simultaneously, but not the homepage.
- **`worldcup/shared.js`** ‚Äî all JS shared across every World Cup page: match-list rendering, ELO color logic, the Rankings view (Rank/Scale), the Knockout bracket tree, and page-view switching. Changes here affect all years simultaneously.
- **`worldcup/fly.js`** ‚Äî the odd one out: unlike the three layers above, this is not inlined into each page's `<style>`/`<script>` block. `build.py` and `build_home.py` each copy it **verbatim** to a `site/`-relative path (`site/fly.js` for the homepage, `site/football/worldcup/fly.js` for every World Cup page and `history.html`) and every page loads it via `<script src="fly.js"></script>` instead. It holds the item-level fly primitive (`flyOutItems`/`flyInItems`, the random-jitter stagger) plus the cross-page click-intercept/`sessionStorage` handoff used by `Home`/`WC YY`/`Tournaments`/`History` navigation ‚Äî shared, verbatim, across the homepage and every World Cup page, which is exactly why it's a linked file rather than a second inlined copy per page. Still no server: a relative `<script src>` resolves fine under `file://` and a static host alike.
- **Inline `<script>` in the page** ‚Äî page-specific setup only: the embedded data constants (`games`, `teams`, `teamElos`), and the page configuration constants (`GAMES`, `YEAR`, `TEAM_ELOS`, `GAMESETS`, `CONFEDERATIONS`, `KNOCKOUT_SIZE`, DOM refs). Nothing else belongs here.

The nav's *markup* (not just its CSS) is also shared, via `worldcup/scripts/nav.py` ‚Äî see that file and `build_home.py` before hand-writing a new nav row anywhere; it likely already covers the row shape you need.

When in doubt: if the change would need to be made in more than one HTML file, it belongs in `worldcup/nav.css`, `worldcup/shared.css`, `worldcup/shared.js`, or `worldcup/fly.js` ‚Äî `nav.css` if it's the fused nav and touches the homepage too, `shared.css`/`shared.js` if it's World Cup pages only, `fly.js` if it's the item-level fly primitive or the cross-page nav protocol (touches the homepage too).

**After editing any of these, run `worldcup/scripts/build.py` before previewing.** None of the pages in `site/` (or `admin/`) link to `nav.css`/`shared.css`/`shared.js` at runtime ‚Äî `build.py` inlines their contents into each page's `<style>`/`<script>` block at build time (see *Shape of the system* above). `fly.js` is the one exception to "inlined," but the same rule still applies in practice: the copy in `site/` is only refreshed by running `build.py`, so editing `worldcup/fly.js` and reloading a page already open from `site/` shows zero change until the next build, exactly like the inlined layers. Editing any of these source files and reloading a page already open from `site/` will show zero change, no matter how correct the edit is ‚Äî the browser is still rendering whatever was embedded (or copied) at the last build.

## A resting-state rule must never outrank a fly-transition rule

`nav.css`'s `.fly-panel`/`.fly-item` classes (and `build_home.py`'s equivalent `.home-group` ones) are how every cross-view and cross-page transition actually moves a chip, row, or card ‚Äî see requirements/navigation.md ‚Üí Transitions.

This used to be one bug waiting to happen: a chip that's mid-fly can simultaneously be `.active` (the outgoing view's selected tab, the arriving page's default-selected tab), and `.view-toggle button.active`/`.strong.active` (nav.css) set their own `transform` (the scale-up "picked up" look). Both the fly classes and the active look shared the single `transform` property, so whichever rule's `transform` won took the *entire* value ‚Äî the losing rule's contribution wasn't blended in, it was discarded outright. The active rule's plain-class specificity happened to beat the fly classes', so an `.active` chip's fly-out/fly-in silently no-opped: it just sat still while every non-active sibling correctly moved. This shipped and went unnoticed for a while because it only reproduces when the flying element happens to also be selected; found via a real report of "two elements get stuck" on the homepage ‚Üí History transition and the History page's default-selected round tab never flying in on arrival. Patched in the moment with `!important` on the fly classes' `transform`.

**Fixed at the root since**, not just patched: `nav.css` and `build_home.py` now use the standalone `translate`/`rotate`/`scale` CSS properties instead of the `transform` shorthand. The chip's resting tilt and `.active`'s "picked up" look are `rotate`/`scale`; the fly classes move via `translate` (plus their own `rotate` for the spin). Because nothing else in either file ever sets `translate`, a fly-transition's distance can no longer be silently overridden by anything, structurally ‚Äî not because of specificity or `!important` discipline, but because there's no other rule left to contend with it. `rotate` is still shared between a chip's resting/selected look and the fly spin, so it keeps `!important` there (see nav.css's comment on `.fly-panel`) ‚Äî but even a future rule that gets that one property's priority wrong would only cost the wrong spin angle mid-flight, never a chip that fails to move at all.

**The rule for any new state class that sets an item's look on a chip** (a new toggle look, a new selected/highlighted variant, anything added to `nav.css`, `shared.css`, or a page template): use `scale`/`rotate` (or `translate`, if it ever needs one), never the `transform` shorthand, on anything that can also carry `.fly-item`/`.fly-panel`/`.home-group.fly-*` ‚Äî that's what keeps it from ever being able to reintroduce this exact bug. If a new rule genuinely needs `transform` for something the individual properties can't express, treat that as the exception, and explicitly verify the new state's chip still flies correctly while it's also the thing flying, not only at rest. One caveat if `set_bracket_game.py`/`shared.css`'s `.bracket-game` pattern is ever extended to have its own active/selected state: it still uses the `transform` shorthand for its resting rotation (`--game-rot`) because nothing there currently competes with it ‚Äî that stops being safe the moment a competing rule is added, and it should get the same `translate`/`rotate` split at that point.

A second, related caveat: the Knockout round toggle's vertical card lift (`shared.js`'s `slideInBracketColumns`/`slideOutBracketColumns`) is a third fly mechanism, separate from `.fly-panel` and `.fly-item` ‚Äî it drives a bracket card's lift/drop directly via inline `style.transform`, not a CSS class. It's currently safe for the same reason as `--game-rot` above (nothing on a bracket card sets a competing resting-state `transform`/`rotate`/`scale` today), but it would reintroduce this exact bug the moment one is added ‚Äî e.g. if a bracket game ever gets its own selected/highlighted look. Any such addition needs the same `translate`/`rotate` discipline as everywhere else, and should switch `slideInBracketColumns`/`slideOutBracketColumns` off the raw `transform` shorthand at that point.

## Architecture boundary

This repo does not depend on other projects or systems on the machine (other codebases, the user's personal planning/documentation system, global tool config) to be built, deployed, or understood ‚Äî everything needed is in this repo. If that ever stops being true (a script starts reading from outside the repo, a doc's meaning depends on external context), that dependency must be called out explicitly wherever it's introduced ‚Äî in the relevant doc, not left implicit.
